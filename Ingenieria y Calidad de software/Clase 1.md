## Que es ingenieria de software?
>[!IMPORTANT]
>Cuando una disciplina puede comenzar a ser medible y repetible entonces se convierte en una ingenieria

   *Disciplina artesanal -> Produccion -> Comercializacion -> ciencia -> Ingenieria*

> Segun IEEE
> La aplicacion de un enforque sistematico, disciplinado y cuantificable al desarrollo, operacion y mantenimiento de software

>Segun SEI
>Ingenieria es la aplicacion sistemica del conocimiento cientifico en la creacion y construccion de soluciones "cost-effective" para resolver problemas practicos al servicio del hombre. La ingenieria en SW es aquella parte de la ingerieria que aplica principios de las ciencias de la computacion y las matematica para alcanzar soluciones "cost-effective" a problemas de sw

# Body of knowledge
Es un requisito para calificar cualquier area de ingeniera como una profesion y describe conocimiento relevante para la disciplina. 
### SWEBOK
El de ingenieria en software es **SWEBOK**. Su proposito es:
- Definir contenido de la ingenieria de software como disciplina
- Promueve comprension a nivel global y consistente
- Clarifica limites entre ingenieria de software con otras disciplinas
- Proporciona fundaments para material de entrenamiento y desarrollo de un curriculo
- Puede soportar certificaicones y licencias de ingenieria en software
- Refleja el estado de desarrollo actual e integra practicas emergentes

#### Problemas, reacciones y causas
| Problema                                     | Reacciones                                                         | Causas                                                       |
| -------------------------------------------- | ------------------------------------------------------------------ | ------------------------------------------------------------ |
| Requerimientos incompletos                   | Dudas sobre lo que me dan y chequeo requerimientos sobre la marcha | Falta de comunicación / expertise                            |
| Descuido / dedazo                            | Pánico / crisis                                                    | Falta de sistema de versionado / falta de entorno de testing |
| No sé dónde ni cómo probar lo que desarrollé | Frustración / dificultad de testeo                                 | Falta de entorno de testing                                  |
| Falta de estándar en código                  | Improvisar / crear estándar / hablar con el equipo                 | Falta de estándares definidos                                |
| Redundancia                                  | Resolver en el momento o escalar al líder                          | Coordinación o comunicación insuficiente                     |
| Estimación de vida útil del producto         | Definir o complementar requerimientos para estimar                 | Requerimientos y entornos cambiantes                         |
Estos problemas son no tecnologicos y tienen mas que ver con temas organizacionales y de contexto.
