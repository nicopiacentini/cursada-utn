Estos problemas son no tecnologicos y tienen mas que ver con temas organizacionales y de contexto.

### Problemas mas comunes de Desarrollo y mantenimiento de software y su impacto
#### Problemas
1. Los proyectos se terminan con mucho atraso o no se terminan directamente
2. Los proyectos exceden el costo estimado una vez finalizados
3. El producto final no cumple con expectativas del cliente
4. El producto final no cumple con requisitos de calidad minimos esperados (como performance, seguridad, escalabilidad)
#### Causas Mas frecuentes
1. No hay administracion y control de proyectos. Es decir, los proyectos no se miden y por ende no se pueden contrastar ni controlar
2. El exito depende de los gurues o expertos y si no estan no funciona
3. Se confunde la solucion con los requerimientos. Esto implica genrar una solucion tecnica que quizas no es la adecuada
4. Las estimaciones son empiricas y no hay historia de proyectos realizados con sus tiempos
5. La calidad es subjetiva y no se puede medir
6. NO se consideran riesgos
7. Se asumen compromisos imposibles de cumplir
8. Las actividades de mantenimiento degradan el software
9. Se comienzan los proyectos con requerimientos carentes, no claros o inestables
#### Reacciones mas frecuentes
1. Buscar un guru que nos saque del problema
2. Agregar gente a un proyecto atrasado
3. Recortar o eliminar:
	1. Actividad que genere documentacion
	2. Etapa de testing
	3. Actividad que no sea codificar
4. Asumir definiciones en lugar de preguntar a usuario
5. Hacer un cambio rotundo con:
	1. Nuevo lenguaje
	2. Nuevo paradigma
	3. Nueva herramienta
6. Justificar el problema delegando la culpa
### Analisis de situacion
Los problemas terminan siendo los mismos y los errores tambien. Entonces se puede decir que el core de estas situaciones suele ser:
>[!IMPORTANT]
>- Cuesta reconocer que existe un problema
>- No se comparten conocimientos
>- Se rechaza la complejidad o se minimiza
>- No se admite la falta de conocimiento
>- Rechazo a la critica constructiva
>- Menosprecio de tareas no tecnicas como documentacion
>- Pensar que existen soluciones que salen por intentar y ser persistente
>- Asumir que producir software lleva esto
