#### Lo suficientemente bueno
A veces se lo considera como un escape o esquive de la calidad. Sin embargo, existen otros factores que juegan con la calidad que son el tiempo y el dinero (Los 3 factores criticos, la calidad, el tiempo y el dinero). Por ende, no se busca el software *rapido, barato y bueno* sino que se negocie con el cliente dichos parametros hasta llegar al exito. Es decir, que se contrasten o comparen estos indices hasta llegar a un software **lo suficientemente bueno**
#### El negocio de factores criticos de exito
Anteriormente estos factores se negociaban con el cliente antes de empezar el proyecto y haciendo estimaciones, para luego optimizar otros parametros que el cliente no tenia en cuenta. Por ejemplo, se negocia con el cliente la funcionalidad, el tiempo y el presupuesto, para luego optimizar la mantenabilidad, portabilidad y disponibilidad del sw. Ahora, no todos los proyectos requieren de tener todos estos parametros maxeados, sino que *se debe negociar con el cliente la suba o baja de alguno de estos hasta encontrar un equilibrio*
#### Lo mejor no siempre es mejor
Producir software hecho especificamente para un contexto lo hace muy apto pero al mismo tiempo muy costoso. Por ende en muchas ocaciones se prefiere un empaquetado que sea mas rapido y menos costoso, pero al mismo tiempo **de menor calidad**
>[! NOTE]
>El software empaquetado puede ser tosco y tener funcionalidades limitadas, según nos comentan nuestros clientes, pero es barato y pueden ponerlo en funcionamiento mañana mismo.

#### Tarde nunca es bueno
En ciertas ocaciones, el cliente desea su producto a un determinado tiempo que es dificil de llegar. Cumplir con las expectativas de calidad y funcionalidad puede ser bueno, pero si tardo demasiado puede hacer que mi cliente se quede con el producto mas rapido. Debo tener en cuenta que migrar de un producto a otro es costoso y no siempre conviene.

#### La cuenta del proyecto
Cada vez que se realize un proyecto, hay que calcular *costos, tiempos de entrega, staff, funcionalidades y calidad en todo su aspecto*. Naturalmente, el cliente debe elegir el nivel de estos pero al mismo tiempo debe saber que no se puede maximizar todo. Debe tener en cuenta **trade-offs** y tomar desiciones. Por ende, **los criterios de exito de un proyecto** estan determinados p.or alcanzar *usar una cantidad X de staff, entregar en Y unidades de tiempo, costar Z pesos, entregar P niveles de funcionalidad y Q cantidad de bugs por funcionalidad*. Normalmente los clientes lo van a rechazar. Por ende es crucial mostrar las consecuencias de acercarse a los parametros que exige, que provoca que bajen otros parametros
